# 1. Аннотация генома (Genome Annotation)

## 1.1. Определение и цели

Аннотация генома — это процесс добавления биологической информации к последовательностям ДНК. Она состоит из трёх основных этапов:

1. **Идентификация участков генома, не кодирующих белки**
2. **Предсказание элементов (генов) на геноме**
3. **Присвоение этим элементам биологической информации (функции)**

Различают **структурную аннотацию** (где находятся гены, CDS, тРНК, рРНК) и **функциональную аннотацию** (что они делают — продукт, семейство, домены). Любая автоматическая аннотация — это гипотеза с определённым уровнем доказательности.

### Лестница доказательств (Evidence Ladder)

```
ORF → coding potential/model → homology hits → domains → curated rules/taxonomy → human sanity
```

| Уровень | Описание |
|---------|----------|
| 🧬 **ORF** | Открытая рамка считывания (последовательность между стоп-кодонами) |
| 📊 **Coding potential** | Оценка статистическими моделями (например, в Prodigal) |
| 🔍 **Homology hits** | Поиск сходства по базам данных (BLAST) |
| 🧩 **Domains** | Обнаружение консервативных доменов (Pfam, InterPro) |
| 📋 **Curated rules** | Применение экспертных правил и таксономической информации |
| 👤 **Human sanity** | Проверка человеком |

## 1.2. Инструменты структурной аннотации

### 1.2.1. Prokka — конвейер для прокариот

Prokka автоматизирует поиск CDS, рРНК, тРНК и выполняет базовую функциональную аннотацию.

**Базовая команда запуска:**

```bash
prokka [опции] <входной_файл.fasta>
```

**Основные опции:**

| Опция | Описание |
|-------|----------|
| `--outdir <дир>` | Папка для результатов |
| `--prefix <преф>` | Префикс для выходных файлов |
| `--kingdom` | Тип организма (Archaea, Bacteria, Mitochondria, Viruses); по умолчанию Bacteria |
| `--cpus <N>` | Количество ядер |
| `--metagenome` | Режим для метагеномных данных |
| `--force` | Перезапись существующей директории |
| `--addgenes` | Добавляет тег `gene` для всех CDS |
| `--proteins <файл>` | Путь к файлу с доверенными белками (например, из EcoCyc) |
| `--evalue <порог>` | Порог E-value для BLAST (например, `1e-9`) |
| `--rfam` | Включить поиск некодирующих РНК через Rfam |
| `--genus`, `--species`, `--strain` | Таксономическая информация |
| `--locustag <преф>` | Префикс для `locus_tag` |
| `--usegenus` | Использовать род-специфичную БД |

**Выходные файлы Prokka:**

| Файл | Описание |
|------|----------|
| `.gff` | Координаты генов и фич в формате GFF3 |
| `.gbk` | Файл GenBank |
| `.faa` | Аминокислотные последовательности всех предсказанных CDS (ключевой для функциональной аннотации) |
| `.ffn` | Нуклеотидные последовательности генов |
| `.txt` | Статистика аннотации |
| `.tsv` | Таблица всех аннотированных фич |
| `.log` | Журнал выполнения |

**Примеры анализа результатов:**

```bash
grep -c ">" файл.faa                    # количество CDS
grep -c "hypothetical protein" файл.faa # количество гипотетических белков
grep -c "ribosomal" файл.faa            # количество рибосомальных белков
```

### 1.2.2. Prodigal — предсказатель генов (внутри Prokka)

Prodigal использует динамическое программирование для оценки потенциальных генов.

**Алгоритм и скоринг:**

1. **Обучение:** анализирует все ORF, учитывая GC-состав (в геномах с высоким GC длинные ORF могут возникать случайно)
2. **Расчёт весов:**
   ```
   Общий вес гена = Вес старта + Кодирующий вес
   ```
   - **Вес старта (start score):** оценивает вероятность, что данный старт-кодон — истинное начало гена (анализ участков связывания рибосом, RBS)
   - **Кодирующий вес (coding score):** основан на частотах дикодонов (гексамерная статистика); представляет собой логарифмическое отношение правдоподобия кодирующей последовательности к некодирующей
3. **Поиск оптимального пути:** динамическое программирование по геному для набора генов с максимальным суммарным весом

**Ключевые параметры точности:**

- Точность стартов для *E. coli* ~96%
- Специфичность (ложноположительные) <5%

**Режим для метагеномов:**

```bash
prodigal -p meta
```

## 1.3. Функциональная аннотация

### 1.3.1. BLAST и E-value

**E-value** — ожидаемое число случайных выравниваний с таким же или лучшим счётом в базе данных заданного размера.

**Формула:**

```
E-value = K × m × n × e^(-λ × S)
```

где:
- `m` — длина запроса (query)
- `n` — общая длина базы данных
- `S` — сырой счёт выравнивания (raw alignment score)
- `K` и `λ` — параметры Карлина-Альтшула (зависят от матрицы замен и штрафов за гэпы)

**Связь с Bit-score (S'):**

```
E-value = m × n × 2^(-S')
```

Типичный порог для уверенной гомологии: **E-value ≤ 1e⁻⁵**

### 1.3.2. KEGG и присвоение KO-номеров

KEGG группирует гены в ортологичные группы (KO). Присвоение KO позволяет связать белок с метаболическими путями.

**Алгоритм BlastKOALA/KAAS:**

1. Поиск гомологов (BLAST/GHOSTX) по базе KEGG GENES
2. Расчёт модифицированной идентичности:
   ```
   Идентичность = pident × min(1, (длина_выравнивания × 2) / (длина_запроса + длина_мишени))
   ```
3. Построение GFIT-таблицы и взвешенная сумма идентичностей для каждой кандидатной KO-группы
4. Присвоение KO на основе взвешенной суммы (опционально — метод бинаправленного лучшего попадания, BBH)

**URL-шаблоны KEGG:**

```
https://www.kegg.jp/entry/<kid>     # общая запись
https://www.kegg.jp/pathway/<kid>   # карта пути
https://www.kegg.jp/module/<kid>    # модуль
https://www.kegg.jp/genome/<kid>    # браузер генома
```

### 1.3.3. Gene Ontology (GO)

GO — словарь функций в трёх осях:

| Ось | Описание |
|-----|----------|
| **Molecular Function** | Что делает белок как активность |
| **Biological Process** | В каком процессе участвует |
| **Cellular Component** | Где находится |

GO-термины позволяют агрегировать функциональную информацию по геному.

### 1.3.4. Другие базы данных

| База | Назначение |
|------|------------|
| **Enzyme Codes (EC)** | Классификация ферментов |
| **InterPro** | Интегрированная база сигнатур белков (домены, семейства) |
| **Protein Data Bank (PDB)** | 3D-структуры белков |

## 1.4. Оценка качества сборки: N50

**N50** — длина контига, при которой сумма длин всех контигов такой же или большей длины составляет 50% от общей длины сборки.

**Расчёт:**

1. Отсортировать контиги по убыванию длины
2. Вычислить общую длину сборки `T`
3. Последовательно суммировать длины, пока сумма не достигнет `0.5 × T`
4. N50 — длина последнего добавленного контига
